# B200 OnPair GPU decode — work summary

Consolidated summary of the OnPair GPU-decode optimization work on the **B200 (Blackwell,
sm_100)** box. This folder summarizes and indexes the work; the detailed source docs it points to
live alongside it in `vortex-cuda/`.

> **Status:** PRELIMINARY. Unlocked clocks (±~5%), single-invocation kernel ranking. **NCU and
> clock-locking are blocked in this container** (`ERR_NVGPUCTRPERM`, no `CAP_SYS_ADMIN`), so the
> limiter is established by *ablation proxies* rather than direct profiling. All shipped kernels
> validated **byte-exact** against CPU decode.

## Environment

- **B200**, Blackwell sm_100, ~148 SMs, 183 GB, CUDA 12.8, nvcomp 5.1.0.21.
- **HGX/PCIe**: Intel Xeon host, **PCIe Gen5 ×16** (~50 GB/s pinned host↔GPU), **C2C disabled** —
  *not* a Grace/GB200 NVLink-C2C system.
- Build: `export PATH="$HOME/.cargo/bin:$PATH"; export LIBCLANG_PATH=/usr/lib/llvm-18/lib` then
  `cargo build -p vortex-bench --features cuda --bin onpair-chunk-bench --release` (~5 min;
  `build.rs` compiles every `kernels/src/*.cu` to PTX).
- **Blocked:** NCU and `nvidia-smi -lgc` clock-locking (need container relaunch with
  `--cap-add SYS_ADMIN`).

## TL;DR

**Shipped (in `pick_auto_kernel`, architecture-gated so GH200 is untouched):** B200 now decodes
**+6 to +46%** over the old `4tpt` default on the big text columns — via two 128-thread kernels,
`split8read_b128o12` (high-`frac_le8` dicts that fit L1) and `b128o12` (everything else),
selected by `pick_general_blackwell(max_entries, frac_le8)`. GH200 keeps its original rule
verbatim (`pick_general_hopper`).

**Per-column B200 decode (fineweb, GiB/s):** bits12 **802** (`split8read_b128o12`, +46% vs old
`4tpt`), bits14 **664** (+9% over `b128o12`), bits16 **637** (`b128o12`).

**The mechanisms (measured):**
1. **Block granularity (128-thread) is the Blackwell lever — not occupancy.** 256→128 threads
   helps; 50%→75% occupancy at fixed 128 buys ~nothing.
2. **8 bytes is the optimal dict read width** (4 B worse, 16 B worse) — the gather is
   transaction/MSHR-bound, not request-width-bound below the 32 B sector.
3. **`split8read` wins exactly while `dict_s8` (entries×8) fits L1** — bits12 (32 KB) and bits14
   (128 KB) win; bits16 (512 KB) ties. **bits14 is the L1-residency sweet spot** (2.3× ratio at
   L1-resident split8read speed). Decode peaks at bits12 and flattens at ~637 for dict ≥256 KB.
4. **bits16 is walled by near-uniform dict access** (top-4096 entries = only ~47% of accesses, so
   no hot subset to cache; the full ~1 MB dict overflows L1).
5. **Whole-decompress (H2D + decode) is transfer-bound** — H2D (~10 GiB/s pageable / ~50 pinned)
   is 60–100× slower than decode, so end-to-end ≈ compression ratio, and the decode-kernel wins
   matter for **on-device decode**, not the H2D-then-decode path.

**The headline discovery (ablation NCU-proxy):** **the byte-ladder *emit* — writing decoded
token bytes into the shared staging buffer — is ~70% of decode runtime** (removing it is 3.4×),
and it is **store-count-bound** (not bank-conflict-bound). Every dict-side optimization tuned the
*second*-biggest cost (~40% gather). The justified-but-unbuilt next step is a **shuffle/`byte_perm`
emit** that cuts store instructions ~25× — see [`02-experiments-tried.md`](02-experiments-tried.md)
and `SESSION_HANDOFF_B200.md` open threads.

## One-line verdict

The dict-gather side of OnPair decode is well-optimized and arch-tuned (split8read + 128-thread
granularity, bits14 sweet spot). The remaining **1.6–3.4× of headroom lives in the emit**, which
is store-count-bound and needs a shuffle-based rewrite — the one change now backed by measurement
rather than guesswork.
